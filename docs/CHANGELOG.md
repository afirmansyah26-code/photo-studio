# Versioning Strategy

Project follows Semantic Versioning.

MAJOR

Breaking changes

MINOR

New functionality

PATCH

Bug fixes

Documentation updates before Version 1.0 may be grouped into the planning milestone as appropriate.

---

# Documentation Policy

Every completed phase must update:

- PROJECT_STATUS.md
- CHANGELOG.md

Documentation is considered part of the implementation.

---

# Commit Policy

Every meaningful commit should have a corresponding changelog entry.

Small internal commits may be grouped into a single release entry.

---

# Release Checklist

Before creating a release:

- Documentation updated
- CHANGELOG updated
- PROJECT_STATUS updated
- Version updated
- Phase completed
- Tests passed
- Build successful

---

# Definition of Done

A release is considered complete when:

- All planned changes are documented.
- Changelog reflects actual implementation.
- Documentation and source code remain synchronized.
- Version number matches the released functionality.
- No undocumented architectural changes remain.
